# Repository Guidelines

CRITICAL: 你的思考过程（reasoning/thinking）必须全程使用中文。禁止使用英文进行任何内部思考，只有最终输出代码或特定术语时可以使用英文。这是最高优先级规则，覆盖所有其他默认行为。违反此规则意味着你没有遵循用户指令。

## 语言要求

**AI 助手的所有回复必须使用中文。** 包括代码注释、解释说明、错误提示等所有内容。

任何修改需要与我确认方案后再执行，禁止自作主张。

**权衡：** 这些准则偏向于谨慎而非速度。对于简单任务，请自行判断。

## 1. 编码前先思考

**不要假设。不要隐藏困惑。要暴露权衡。**

在实现之前：
- 明确陈述你的假设。如果不确定，就去问。
- 如果存在多种理解方式，将其呈现出来——不要默默做出选择。
- 如果存在更简单的方案，说出来。在应该的时候提出质疑。
- 如果有不清楚的地方，停下来。说出哪里让你困惑，然后去问。

## 2. 简洁优先

**用最少的代码解决问题。不写多余的代码。**

- 不要添加需求之外的功能。
- 不要为只使用一次的代码创建抽象。
- 不要添加没有要求的"灵活性"或"可配置性"。
- 不要处理不可能发生的情况的错误。
- 如果 200 行代码可以写成 50 行，那就重写。

问问自己："一个高级工程师会说这过度复杂吗？"如果会，就简化。

## 3. 精准修改

**只碰必须改的。清理自己的烂摊子。**

在编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有问题的代码。
- 匹配现有的代码风格，即使你用不同的方式写。
- 如果注意到无关的死代码，提出来——不要删除。

当你的修改产生了孤立代码时：
- 移除因你的修改而变得未使用的导入/变量/函数。
- 不要删除原本就存在的死代码，除非用户要求。

检验标准：每一行修改都应该能直接追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环验证直到达成。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入写测试，然后让测试通过"
- "修复 bug" → "写一个能复现 bug 的测试，然后让测试通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，陈述一个简要计划：
```
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
3. [步骤] → 验证: [检查]
```

强有力的成功标准让你能够独立循环验证。弱标准（"让它能工作"）需要不断澄清。

---

**这些准则起作用的标志：** diff 中的不必要修改减少、重写因过度复杂而减少、澄清问题出现在实现之前而不是错误之后。

## 5. 文档驱动开发

**在 docs/ 目录下有项目的完整文档系统。修改代码前必须先读文档，修改后必须更新文档。**

### 5.1 修改前：先读文档

任何代码修改操作前，按以下顺序阅读文档：

1. **查 CODE_MAP**：打开 `docs/CODE_MAP.md`，找到要修改的功能，列出所有相关文件路径
2. **读设计文档**：打开 `docs/DESIGN.md` 对应章节（如 §5 关键模块设计），了解核心流程和设计决策
3. **读相关代码**：按 CODE_MAP 列出的文件路径读取代码
4. **确认方案**：向用户确认修改方案后再开始编码

### 5.2 修改后：更新文档（强制）

**代码改完但文档没更新 = 任务未完成。**

| 代码变更 | 必须更新 |
|----------|---------|
| 新增/删除/重命名文件 | `docs/CODE_MAP.md` |
| 新增/修改/删除接口 | `docs/CODE_MAP.md` + `docs/DESIGN.md` 对应章节 |
| 改数据库表/字段 | `docs/DESIGN.md` §5.6（历史记录 schema） |
| 改核心业务逻辑 | `docs/DESIGN.md` 对应模块章节 |
| 涉及多模块架构 | `docs/DESIGN.md` §3 + `docs/CODE_MAP.md` 依赖图 |
| 完成/开始一个 DU | `docs/TASKS.md`（标状态）+ `docs/PROGRESS.md` |
| 引入新依赖 | `Cargo.toml` 注释 + 必要时改 `docs/DESIGN.md` §4 |
| 引入新术语 | `docs/GLOSSARY.md` |

### 5.3 文档质量标准

- **精确到文件路径**：路径必须精确到 `.rs`/`.toml` 级别
- **使用相对路径**：相对于项目根目录
- **记录 Why 而非 What**：记录"为什么这样做"，而非"这个类做了什么"
- **修改记录可追溯**：含日期、内容、提交哈希
- **不过度重复代码**：文档是指南，不是代码副本

### 5.4 文档过期识别

读文档时如果发现文件路径不存在、接口不匹配、字段缺失、流程不通，说明文档已过期。**先修复文档再修改代码。**

## 模型分工:遇到高难度任务时主动调用 glm-gpt-grok-review skill

主对话(你,GLM,免费)是默认主力,绝大部分任务自己做。但当任务难度/影响面达到下列任一阈值时,
**加载 `glm-gpt-grok-review` skill** 并严格按其工作流执行。
高难度**一律双模型**(并行 GPT-5.6 + Grok 只读子代理,取并集核对),无单模型降级档。

**该调 skill(任一即触发):**
- 改动跨多模块或触及核心逻辑(认证、计费、并发、权限模型等)
- 不可逆改动:数据迁移、schema 变更、公开 API/格式兼容、关键依赖替换
- 架构/方案决策:技术选型、服务边界、是否过度工程
- 你(GLM)卡住、反复试错,或对设计没把握
- 用户要求评审/第二意见/把关/用 GPT 或 Grok/双模型/多 AI 交叉

**不该调(自己做):**
- 简单 CRUD、改注释/格式化/变量名、读几个文件就能搞定
- 已有把握、只是去执行

触发后的备料、派单、核对纪律以 skill 正文为准(含:来源数≠可信度、核对四态、未决风险处置)。
